01Context & goals

Part 1 shipped the base app: built-in markdown topics, checklist progress, gamification (coins/XP, badges, daily goal, confetti, reward shop), notes, export/import, Pagefind search, PWA, and VPS deploy. All state lives in one localStorage key (levelup:data).

Part 2 adds three capabilities without removing built-in topics:

1 · Upload topics

Upload a topic as one .md file; it's parsed and saved to browser storage, then behaves like a built-in topic.

2 · Custom rewards

A full editor (add / edit / delete). The current 6 become deletable seeded defaults with a reset.

3 · Search modal

Search becomes a popup (Cmd/Ctrl+K) showing results inline; the /search page is removed.

Plus a documented markdown format (§4) shipped as a help doc, an in-app template download, and a link — so the user knows how to author an uploadable topic.
02Architecture shift — client-rendered user content

Built-in topics are static (built at deploy time). User uploads can't be statically built, so user topics are parsed and rendered client-side from storage. The store's in-memory content index becomes the union of both sources:

built-in index
+
user topics
→
merged ContentIndex
→ progress · coins · badges · stats (unchanged)

Because progress.ts and gamification.ts are generic over a ContentIndex keyed by topicSlug/lessonSlug, everything works across both sources with no logic changes. User-topic ids are prefixed u-<slug>-<rand> so their keys never collide with built-in slugs.

04Topic markdown upload format

One .md file = one topic. Top YAML frontmatter is the topic; each # H1 starts a subtopic; a ## Checklist task list under a subtopic defines its checklist; everything else under the H1 is that subtopic's theory.

---
title: My Topic
icon: "📦"
description: One-line summary shown on the card.
---

# First Subtopic

Theory markdown. Supports code blocks, > [!NOTE] callouts, and ```mermaid diagrams.

## Checklist
- [ ] First thing to learn
- [ ] Second thing to learn

# Second Subtopic

More theory...

## Checklist
- [ ] Another item
RuleDetail
Frontmattertitle required; icon, description optional (flat key: value).
SubtopicsEach # H1 = one subtopic; text = title; order = document order.
ChecklistThe task list after a ## Checklist heading; that section is stripped from the rendered body.
Minimums≥1 subtopic per topic; a subtopic may have no checklist (renders body only).

Parser src/lib/parseTopic.ts (pure, unit-tested) → { topic, errors[] }: parse flat frontmatter, split by # headings, extract & strip ## Checklist, slugify titles, generate u-<slug>-<rand> id, and return clear validation messages.

05Client-side rendering of user content

src/lib/renderMarkdown.ts using marked (new dep), reusing Part 1 assets:

  • Mermaid — ```mermaid fences → <pre class="mermaid"> so the existing Mermaid.svelte renders them.
  • Callouts — > [!NOTE|TIP|WARNING|CAUTION|IMPORTANT] → existing .markdown-alert markup/CSS.
  • Code highlighting — lazy-loaded Shiki (github-light/github-dark, emits .astro-code), loaded only on a user lesson view.

User views are client-only (content is in storage, absent at build): one static page src/pages/u.astro hydrates UserTopicView.svelte with client:only="svelte", routing by query:

URLView
/u?t=<topicId>Subtopic list for that topic.
/u?t=<topicId>&l=<lessonSlug>Lesson: rendered body + reused Checklist + Notes.
06Store / index integration
  • New pure helper userTopicsAsIndex(userTopics) → TopicMeta[] (id as slug, lessons → checklistLen).
  • ensureIndex() returns the merged index = built-in fetched + userTopicsAsIndex; recomputed when userTopics changes.
  • New actions: addUserTopic, deleteUserTopic (also purge its progress/notes keys), addReward, updateReward, deleteReward, resetRewards.
  • StatsSummary overall % uses the merged index so uploaded topics count too.
07Custom rewardsnew
  • src/data/rewards.ts keeps DEFAULT_REWARDS (current 6) + WEEKLY_ALLOWANCE_CAP; the shop stops importing the list directly.
  • RewardShop.svelte reads $appData.rewards; RewardRow/RedemptionList resolve metadata from it (fallback to id).
  • New RewardEditor.svelte on /rewards: list with edit/delete, an "Add reward" form (name, emoji, coin cost, rupiah — validated positive), and "Reset to defaults". Custom rewards persist and survive export/import.
